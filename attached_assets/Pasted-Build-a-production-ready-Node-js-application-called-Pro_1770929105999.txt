Build a production-ready Node.js application called "ProtectiveShell".

ProtectiveShell is an internal monitoring dashboard for backup outcomes and infrastructure health. It must run on Replit for development and be easily deployed later to an Ubuntu/Debian VM (where NetBird will be installed). The code must NOT depend on Replit-specific features.

========================
HIGH-LEVEL PURPOSE
========================
ProtectiveShell provides:

1) Backup Job Monitoring via IMAP email notifications (MXroute mailbox):
   - Supported systems ONLY: Veeam, Proxmox Backup Server (PBS), Synology backup notifications.
   - Poll IMAP on a schedule (default hourly).
   - Classify emails as success/warning/failure for a job.
   - Detect missing expected email (job ran but no email arrived by deadline).
   - Daily summary report email.
   - Immediate alerts for failed/missing (warnings optional).

2) Proxmox Host Health Monitoring via agentless SSH (username/password):
   - Add Proxmox hosts in UI.
   - Poll hosts on a schedule (default hourly).
   - Detect storage setup and health:
     - ZFS: pool degraded, checksum errors, resilvering.
     - mdadm: degraded array, rebuild.
     - SMART: drive health warnings/errors.
     - Dell RAID: if storcli exists, detect degraded VD/PD predictive fail/rebuild.
   - Normalize to status: OK / WARN / CRIT / UNKNOWN.
   - Show a dashboard status widget and a dedicated Proxmox Health page.
   - Provide “Test Connection” and clear “needs action” messages (auth failed, sudo needed, tool missing, etc).

3) Mailbox Retention:
   - Automatically delete processed IMAP emails older than 7 days to save MXroute storage.
   - Only delete emails after they are safely ingested in DB (ingested_ok = true).
   - Expunge after deletion.

IMPORTANT: This MVP must be deterministic and reliable. Do NOT implement AI classification yet. Unmatched/unknown emails should be flagged for review in the UI.

========================
RELIABILITY REQUIREMENTS (MUST)
========================
1) IMAP must be stateless polling:
   - Each poll run: connect -> fetch new messages -> process -> disconnect.
   - Do not keep long-lived IMAP sockets.

2) IMAP checkpointing:
   - Track per-folder UIDVALIDITY and last_seen_uid in the database.
   - If UIDVALIDITY changes, reset safely without duplicating or missing messages.

3) Idempotent ingestion:
   - emails table must enforce uniqueness on (folder, uidvalidity, uid) AND also message_id when present.
   - Worker must handle duplicates gracefully.

4) Expectation engine for "missing email" detection:
   - Implement an expected_runs table and logic that generates expected runs from job schedules.
   - evaluateMissing.js must mark runs as MISSING when now > deadline_at AND no linked event exists.

5) Timezone correctness:
   - Use TIMESTAMPTZ in PostgreSQL for all timestamps.
   - Add APP_TIMEZONE env var and use it for schedule calculations and daily report timing.

6) Worker concurrency safety:
   - Use row-level locking where needed (SELECT ... FOR UPDATE SKIP LOCKED) for any worker queue operations.
   - Ensure web and worker don’t corrupt state; prefer idempotent updates with updated_at checks.

7) Strict timeouts:
   - SSH checks must time out per host (10 seconds).
   - One dead host must not hang the entire loop.
   - Record UNKNOWN on timeout, and create an "unreachable" incident only after N consecutive failures (default 3).

8) Prefer machine-readable output:
   - For SMART, use smartctl -j and parse JSON (do NOT grep text when JSON exists).
   - For ZFS/mdadm/storcli, use stable signals; if parsing is uncertain, set UNKNOWN (never assume OK).

9) Fail-safe behavior:
   - Any uncertain parse yields UNKNOWN or NEEDS_REVIEW, not OK.
   - Log errors clearly, but never log secrets.
