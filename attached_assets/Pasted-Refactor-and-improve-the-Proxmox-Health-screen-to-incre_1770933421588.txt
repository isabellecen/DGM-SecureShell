Refactor and improve the Proxmox Health screen to increase operational visibility and decision-making clarity.

The current UI only shows a high-level status (OK / Warning / Unknown), which is not sufficient.

We need the Proxmox Health dashboard to show actionable health signals at a glance.

========================
GOAL
========================
The Proxmox Health screen must immediately answer:

"Do I need to touch this host? Why?"

It should surface storage health details directly on the host cards without requiring deep navigation.

========================
REQUIREMENTS
========================

1️⃣ Enhance Proxmox Host Card (Summary View)

Each host card must now display:

A) Storage Type Detected
   - ZFS
   - mdadm
   - Dell RAID
   - Mixed
   - Unknown

B) Health Summary (1–3 short lines max)

Examples:

- Storage: ZFS
- Pools: 2 total (1 DEGRADED)
- Disks: 1 SMART warning

OR

- RAID: Virtual Disk DEGRADED
- SMART: 1 predictive failure

OR

- Monitoring Error: SSH timeout
- Consecutive Failures: 3

C) Component-Level Indicators
If applicable, show small badges or inline labels for:
- ZFS
- SMART
- RAID
- mdadm

D) Last Check timestamp
E) Consecutive Failures count (if > 0)
F) Quick action button:
   - "Run Check Now"

Do NOT overload the card. Keep it compact but informative.

========================
DATA MODEL UPDATE
========================

Modify proxmox_checks.payload_json structure to normalize component health clearly.

Instead of a raw dump, use structured JSON like:

{
  "overall_status": "OK | WARN | CRIT | UNKNOWN",
  "storage_type": "ZFS | MDADM | RAID | MIXED | UNKNOWN",
  "components": {
    "zfs": {
      "status": "OK | WARN | CRIT",
      "pools": [
        { "name": "rpool", "state": "ONLINE" },
        { "name": "data", "state": "DEGRADED" }
      ]
    },
    "smart": {
      "status": "OK | WARN | CRIT",
      "disks_total": 8,
      "disks_warning": 1,
      "disks_failed": 0
    },
    "raid": {
      "status": "OK | WARN | CRIT",
      "virtual_disks_degraded": 1,
      "predictive_failures": 1
    }
  },
  "monitoring_error": null
}

If there is a monitoring failure (SSH timeout, auth error, sudo issue), use:

"monitoring_error": "SSH_TIMEOUT | AUTH_FAILED | SUDO_DENIED | TOOL_MISSING"

And set overall_status = UNKNOWN.

========================
CARD DISPLAY LOGIC
========================

Implement display logic:

- If monitoring_error exists:
    Show:
    "Monitoring Error: SSH timeout"
    Show consecutive_failures
    Set card color to neutral/gray

- If overall_status = CRIT:
    Show top 1–2 critical issues only
    Red status badge

- If overall_status = WARN:
    Show top warnings (max 2)
    Yellow badge

- If OK:
    Show:
    "All storage healthy"
    Green badge

========================
DETAIL PAGE ENHANCEMENT
========================

Update the Proxmox Host Detail page to include:

Tabs:
- Overview
- Disks
- Pools/Arrays
- History

Overview tab:
- Storage Type
- ZFS pool table (if applicable)
- RAID summary (if applicable)
- SMART summary

Disks tab:
- Disk list table:
  - Name
  - Model
  - SMART Status
  - Reallocated sectors
  - Pending sectors
  - Temperature

Pools/Arrays tab:
- ZFS pools OR mdadm arrays OR RAID VDs
- State
- Resilver/Rebuild status

History tab:
- Timeline of last 20 checks
- Status transitions

========================
FAIL-SAFE RULE
========================

If parsing fails or output is unclear:
- Do NOT mark as OK.
- Mark component as UNKNOWN.
- Reflect that in card.

========================
UX REQUIREMENTS
========================

- Maintain existing dark theme style.
- Do not clutter UI.
- Keep card height consistent across hosts.
- Prioritize clarity over verbosity.

DELIVERABLE
========================

- Updated Proxmox Health page
- Updated host detail page
- Updated payload normalization logic in proxmoxHealth.js
- Clean helper functions for status summarization
- No breaking changes to existing schema outside payload_json

Do not remove existing functionality. Improve clarity and operational visibility only.
